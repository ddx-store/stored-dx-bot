## Core idea

I already have:
- A Telegram bot
- A Gmail inbox
- Duck email aliases forwarded into my main Gmail
- OTP emails already arrive in Gmail successfully

I want a new feature where I can send the bot:
- email
- secret/password

1. Receive the request from Telegram
2. Start account creation on my website
3. If my website triggers OTP verification by email, automatically wait for the OTP email in Gmail
4. Read the OTP from a specific Gmail label or filtered inbox
5. Submit the OTP back to my website automatically
6. Confirm that the account is verified / logged in / ready
7. Send the final result back to Telegram

This must work ONLY for my own website and must be designed for authorized internal use.

Use direct API integration with my website backend if possible.

- preferred: HTTP API client
- optional fallback: Playwright browser client

## Main workflow

Telegram Bot
-> receives email + password/secret
-> creates a job
-> calls my site integration layer
-> starts registration
-> waits for OTP email in Gmail
-> reads the matching OTP message
-> extracts the OTP code
-> submits the OTP to my site
-> confirms success or failure
-> sends a final status message on Telegram

### 2) Job orchestration
Implement a clean job manager with persistent state.

- pending
- creating_account
- waiting_for_otp
- verifying_otp
- completed
- failed

Track:
- job_id
- email
- current status
- created_at
- updated_at
- error_message
- otp_attempts
- final result

### 3) Gmail OTP watcher
Use Gmail API, not IMAP.
Read from a specific Gmail label dedicated to OTP emails.

Requirements:
- fetch unread or recent emails from the configured Gmail label
- detect the correct email for the current job
- filter by sender, subject pattern, timestamp, and intended recipient when possible
- extract OTP code or activation link from the email body
- mark processed emails to avoid duplicate reuse
- support polling with retry + timeout
- structure the code so push notifications can be added later if needed

Important:
Multiple jobs may run close together, so the OTP matching logic must avoid mixing codes between jobs.
Use:
- target email address
- time window
- known sender
- subject filters
- recent message ordering
- message deduplication

### 4) Website integration layer
Build a provider-style abstraction for the website integration.

Create an interface like:
- create_account(email, password)
- request_otp(email)
- submit_otp(email, otp)
- finalize_account(email)
- get_account_status(email)

Provide:
1. API-based implementation as the primary method
2. optional browser-based implementation stub or fallback using Playwright, isolated in a separate module

Do not hardcode my site logic everywhere.
Keep the site integration replaceable and modular.

### 5) OTP extraction
Implement a parser that can extract:
- numeric OTP codes
- activation links
- fallback patterns if the email format changes slightly

Support regex-based extraction with configurable patterns.
